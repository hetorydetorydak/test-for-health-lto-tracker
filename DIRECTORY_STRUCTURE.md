**Business Rule Execution:**

Each file contains JavaScript functions that execute within ServiceNow's Rhino JavaScript engine:

- Triggered by table events (insert, update, delete)
- Full access to GlideRecord API
- Can query and modify other tables
- Runs server-side (not in browser)

## Naming Conventions by Directory

### `/src/fluent/tables/`

Files follow pattern: `[table_name].now.ts`

**Example:**

- `license.now.ts` → exports table `x_1998335_testlto_license`
- `facility.now.ts` → exports table `x_1998335_testlto_facility`

### `/src/server/`

Files follow pattern: `br_[table_name].js` or `[utility_name].js`

**Example:**

- `br_license.js` → Business rules for license table
- `br_facility.js` → Business rules for facility table
- `scheduled_alert_job.js` → Scheduled job implementation

**Example:**

- `ExpiringLicensesTable.jsx` + `ExpiringLicensesTable.css`

## Summary

This project uses a **structured, layered architecture**:

1. **Metadata Layer** (`src/fluent/`) — Define what (tables, roles, acls)
2. **Logic Layer** (`src/server/`) — Implement how (business rules, jobs)
3. **UI Layer** (`src/client/`) — Present data (React components)
4. **Configuration** (Root) — Connect & document everything

**Key Principle:** Everything is organized by **concern** (tables, roles, business rules) not by **type** (all .js files together).

This makes the codebase scalable, maintainable, and easy for AI assistants to navigate and extend.
